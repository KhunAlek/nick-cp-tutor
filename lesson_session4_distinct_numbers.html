Distinct Numbers — Removing Duplicates the Easy Way (and Its One Weak Spot)

Last time: Repetitions — streak pattern, correct reset on the first try Today: a tool that deletes duplicates for you, and the one case where it needs backup
Open Distinct Numbers on CSES ↗

Today's problem: how many different numbers are there

Today's suggested problem, Distinct Numbers, gives you a list of numbers and asks: ignoring repeats, how many different values are actually in there?

For example, if the list is 2 3 2 5 3 2, the numbers 2, 3, and 5 each show up, just some of them more than once. The answer is 3 — not 6.

The first idea: a set

You could solve this by checking, for every number, whether you've already seen it before — but that means a loop inside a loop, checking the whole list again and again. Python has a shortcut: a set.

A set works like a list, except the moment you put a duplicate into it, the duplicate just doesn't count. Wrap any list in set(...) and every repeat disappears automatically. Then len(...) counts what's left.

A set keeps only one copy of each value. Wrapping a list in set(...) is an instant duplicate-remover.

The one weak spot — and the fix that has no weak spot

A set is fast because it uses a shortcut code (a hash) to decide where each number goes, like a post office sorting letters into cubbies by zip code. Almost always that spreads numbers out evenly and stays fast. But CSES includes a stress test built from numbers specifically chosen to crowd into the same cubby — and on that one test, a plain set() can time out even though the logic is completely correct.

A set is normally fast because numbers spread out. A very specific, rare pattern of numbers can defeat that.

The fix that has no weak spot: sort the numbers first. Sorting never has an unlucky case — same time no matter what order the numbers came in. Once sorted, every copy of the same number stands right next to its twin, so you can walk through once and count every time the number changes from the one right before it.

You already did almost exactly this

In Repetitions, you tracked when a letter changed to know a streak had broken. This is the same walk-through-and-compare-to-the-last-one idea — instead of tracking how long a run is, you're counting how many runs there are in total.

Why this stops short on purpose

The new idea today is sorting-then-counting-changes, not the loop mechanics — you've written this exact walk-and-compare shape before, in Repetitions. Paste just the right-hand block on its own and run it: it reads and sorts the numbers, then stops, printing nothing. That silence means the counting loop is still left for you to write.

Common mistakes on this one

✕
Using plain set(numbers) and stopping there — correct on almost every test, but has a real, documented weak spot CSES tests for on purpose.
✕
Starting the count at 0 instead of 1 — the very first number in the sorted list is already one distinct value, even before any comparison happens.
✕
Comparing numbers[i] to numbers[i+1] instead of numbers[i-1] — an easy direction mix-up that shifts every comparison by one.
✕
Forgetting to sort before comparing neighbors — comparing unsorted neighbors tells you nothing about duplicates.

Hints, if you want them

Try it yourself first. If you're stuck, open these one at a time — don't jump straight to the last one.

Hint 1
If two numbers are the same value, and the list is sorted, where would they end up sitting relative to each other?
Hint 2
What should your count start at before the loop even begins — 0, or 1? Think about whether the very first number in the list is already "one distinct value" on its own.
Hint 3
Loop from index 1 to the end. At each index i, compare numbers[i] to numbers[i-1]. If they're different, you've found a new distinct value — add 1 to your count.

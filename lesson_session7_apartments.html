A new shape of problem: matching two separate groups

Every problem so far has given you one list to think about. Today's problem, Apartments, gives you two lists that need to be paired up. One list is a group of applicants, each wanting an apartment close to a certain size. The other list is the apartments that actually exist, each with its own size. Your job: match as many applicants to apartments as you can, where "close enough" means the sizes differ by at most some allowed amount, k. Each apartment can only go to one applicant, and each applicant only gets one apartment.

Why checking every pair is too slow

The obvious idea: for every applicant, scan through every apartment to find one close enough that isn't already taken. With up to 200,000 of each, that's up to 40 billion comparisons. Far too slow.

Wrong — correct, but far too slow
n, m, k = map(int, input().split()) desired = list(map(int, input().split())) apartments = list(map(int, input().split())) used = [False] * m count = 0 for i in range(n): for j in range(m): if not used[j] and abs(apartments[j] - desired[i]) <= k: used[j] = True count += 1 break print(count) # gives the right answer on small tests # but with 200,000 of each, this checks # up to 40 billion pairs. Times out.
Right — sort first, then walk once
n, m, k = map(int, input().split()) desired = list(map(int, input().split())) apartments = list(map(int, input().split())) desired.sort() # sort both lists first -- the new idea today apartments.sort() i = 0 j = 0 count = 0 # ...loop while i < n and j < m. Compare apartments[j] to desired[i]: # if the apartment is too small (apartments[j] < desired[i] - k), move # j forward. If it's too big (apartments[j] > desired[i] + k), move i # forward. Otherwise it's close enough -- count it, and move BOTH i # and j forward. You already know how to write a while loop with an # if/elif/else inside -- write that part yourself.
The new tool: two pointers

i and j are called pointers — they're just plain variables that hold a position in a list, exactly like the index you already use in for i in range(len(s)). What's new is having two of them, each walking through its own sorted list, and only ever moving forward — never back. Paste just the right-hand block on its own and run it: it sorts both lists and sets up the pointers, then stops, printing nothing. That silence means the walking loop is still left for you to write.

How the two pointers actually move

Sort both lists first — this step matters as much as the loop itself. Then at every step there are exactly three possibilities, using the real Apartments sample: desired sizes 45, 60, 60, 80 (sorted), apartments 30, 60, 75 (sorted), k = 5.

Every step does exactly one comparison and moves exactly one or two pointers forward. Nothing is ever revisited.

Common mistakes on this one

✕
Forgetting to sort BOTH lists before the loop — a two-pointer loop on unsorted lists doesn't crash, it just quietly gives a wrong number.
✕
Mixing up which pointer moves for "too small" vs "too big" — it's easy to swap i and j by accident.
✕
Forgetting to move BOTH pointers on a match — moving only one means an apartment or applicant could get reused.
✕
Using strict </> instead of checking against desired[i] - k and desired[i] + k exactly — an off-by-k error here silently rejects valid matches.

Hints, if you want them

Try it yourself first. If you're stuck, open these one at a time — don't jump straight to the last one.

Hint 1
If your answer keeps coming out as 0, what's the very first thing you'd want to check about the two lists before the loop even starts?
Hint 2
At each step, you're comparing one apartment to one desired size. What are the only three things that comparison could tell you — and does each of those three need a different pointer to move?
Hint 3
Print i, j, desired[i], and apartments[j] once per loop iteration. Watching them change step by step — which one moves, and when — usually makes it clear which branch is behaving unexpectedly.

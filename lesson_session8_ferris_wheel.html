Ferris Wheel — Two Pointers Starting From Opposite Ends

Last time: Apartments — two pointers marching the same direction Today: the same tool, pointers starting far apart and closing in
Open Ferris Wheel on CSES ↗

A twist on the same idea

In Apartments, both pointers started at the beginning of their lists and only ever moved forward, one step at a time. Today's problem, Ferris Wheel, uses two pointers again — but this time they start at opposite ends of the same sorted list and walk toward each other in the middle. Same tool, different direction. You'll recognize the shape immediately.

The problem in plain words

There are some kids waiting for a Ferris wheel. Each kid has a weight. Each gondola can carry at most 2 kids, and the two kids in a gondola together can't weigh more than some maximum x. You want to use as few gondolas as possible to get everyone on the ride.

Why checking every pair is too slow again

The obvious idea: pick a kid, then scan through everyone else left over to find someone they can share a gondola with. That's correct, but for every kid you might scan the whole remaining list — with up to 200,000 kids, that's the same 40-billion-checks problem from Apartments.

Wrong — correct, but far too slow
n, x = map(int, input().split()) weight = list(map(int, input().split())) used = [False] * n count = 0 for i in range(n): if used[i]: continue used[i] = True count += 1 for j in range(n): if not used[j] and weight[i] + weight[j] <= x: used[j] = True break print(count) # right idea, but for every unmatched kid # it can scan the whole list again. # Too slow once n gets large.
Right — sort, then close in from both ends
n, x = map(int, input().split()) weight = list(map(int, input().split())) weight.sort() # sort first -- always pair the lightest with the heaviest left = 0 right = n - 1 count = 0 # ...loop while left <= right. If left == right, that's one kid left in # the middle with no partner -- count them and stop. Otherwise: check # if weight[left] + weight[right] <= x. If they fit, move left forward # too (both got seated). Either way, count one gondola and move right # inward -- the heaviest kid always leaves one way or another, either # paired up or riding alone. You already know how to write a while # loop with an if/else inside from your Stage 1 projects -- write # that part yourself.
Why this stops short on purpose

The new idea today is pairing lightest-with-heaviest, not the loop mechanics — you've written while loops with if/else before. Paste just the right-hand block on its own and run it: it sorts the weights and sets up the pointers, then stops, printing nothing. That silence means the closing-in loop is still left for you to write.

The idea: pair the lightest with the heaviest

Sort everyone by weight first. Now think about the single heaviest kid left. If they can share a gondola with anyone at all, it's the lightest kid left — nobody else gives them a better chance of fitting under the limit.

Two outcomes, every time

If the lightest and heaviest fit together (their weights add up to x or less), put them in one gondola together — that's one gondola for two kids, and both pointers move inward. If they don't fit, the heaviest kid has to ride alone — that's still one gondola, but only the heaviest end moves inward, since the lightest kid might still pair with someone else.

Weights sorted: 2, 3, 7, 9. The lightest and heaviest don't fit together, so the heaviest takes a gondola alone.

Walking the full CSES sample

Same weights, x = 10. Watch how many gondolas it takes and which pointer moves at each step.

Three steps, three gondolas — every kid seated, no wasted capacity.

Common mistakes on this one

✕
Forgetting to sort weight before setting up the pointers — the lightest-with-heaviest logic only works on a sorted list.
✕
Not handling left == right as its own case — that's the one leftover kid in the middle, easy to count twice or not at all.
✕
Only moving left forward when a pair fits, but forgetting right always moves inward either way.
✕
Forgetting to count a gondola when the heaviest rides alone — every gondola counts, paired or not.

Hints, if you want them

Try it yourself first. If you're stuck, open these one at a time — don't jump straight to the last one.

Hint 1
If the lightest and heaviest kid don't fit in a gondola together, does the heaviest kid have any chance of fitting with anyone else on the list?
Hint 2
What should happen to right when the pair fits, versus when it doesn't? Is there a step that should happen either way, regardless of whether they fit?
Hint 3
Check what happens the moment left == right. What does that mean about how many kids are left, and does your loop handle that case before trying to compare weight[left] with itself?

A trick just for this one

If your gondola count comes out too high or too low, print left, right, and count once per loop. Watch for the moment left == right — that's the single leftover kid in the middle, and it's easy to accidentally count them twice or not at all if that case isn't handled separately.

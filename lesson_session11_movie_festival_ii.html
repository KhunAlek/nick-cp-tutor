The problem in plain words

The problem in plain words

Same festival, same n movies with start and end times — but now there are k screens instead of just one, all showing movies at the same time. What's the biggest total number of movies that can be watched, across all k screens, with none overlapping on the same screen?

Your Movie Festival code already solves this for k = 1. The question today is: what has to change when you have several screens instead of one?

The trap: reusing last session's code unchanged

It's tempting to paste your Movie Festival solution in as-is — it still runs, and it still prints a number. But it quietly ignores every screen except one, so it undercounts as soon as k is more than 1.

Treating k screens as if there were only one is the single most common way to get this one wrong.

The fix: track k screens with a heap

Sort movies by start time this time (not end time — you need to consider movies in the order they'd actually be offered to a screen). Keep a collection of "when does each currently-used screen next become free" — up to k of them. For each movie, check the screen that frees up soonest: if it's free in time, reuse it; otherwise, open a new screen if you have fewer than k in use; otherwise, skip the movie.

Python's heapq module is a new tool for exactly this: it keeps a list where you can always pull out the smallest value quickly, without re-sorting the whole thing every time. heapq.heappush(screens, value) adds a value; heapq.heappop(screens) removes and returns the smallest one.

A heap keeps the "frees up soonest" screen one heappop away, instead of scanning every screen each time.

Why this stops short on purpose

The new tool today is heapq itself — the loop shape (walk a sorted list, keep a running count) is something you've written many times. Paste just the right-hand block on its own and run it: it imports heapq, sorts the movies, and sets up an empty screens list — then stops, silently, with nothing printed. That silence is the loop still waiting to be written.

Common mistakes on this one

✕
Reusing the single-screen Movie Festival logic unmodified — it happily runs and prints a number, just the wrong one.
✕
Sorting by end time like last session, instead of start time — this problem needs movies considered in the order screens would actually be offered them.
✕
Forgetting to push a screen's time back onto the heap when it turns out not to be free yet — that screen quietly disappears from consideration.
✕
Opening a new screen even when k are already in use — always check the count of screens in use against k before adding another.

A trick just for this one

If your count looks wrong, print the screens list right after each movie you process. It should never have more than k numbers in it. If it's growing past k, you're opening new screens when you should be reusing one.
